# Примеры проектов

Эта директория содержит два минимальных демонстрационных сервиса, которые иллюстрируют использование **cloud-native-cicd-template** в реальных условиях.

Примеры намеренно сделаны универсальными и не содержат реальных названий проектов, доменов, учетных данных или значений, специфичных для инфраструктуры.

---

## 📁 Содержимое

```text
example/
  backend-service/
  frontend-service/
```

### backend-service

Минимальный пример backend-сервиса, включающий:

- Dockerfile
- Файлы значений Helm
- Легковесные обертки для workflows:
    - deploy
    - rollback
    - uninstall

### frontend-service

Минимальный пример frontend-сервиса, включающий:

- Dockerfile
- Простая статическая сборка
- Файлы значений Helm
- Легковесные обертки для workflows:
    - deploy
    - rollback
    - uninstall

### Пример интеграции PostgreSQL

Оба сервиса (backend и frontend) могут быть расширены для интеграции с PostgreSQL. Директория `helm/web-service/templates` содержит шаблоны для PostgreSQL, которые можно включить, обновив файл `values.yaml` в директории `deploy/` каждого сервиса.

## 🎯 Цель

Цель этих демонстрационных сервисов — показать:

- Как подключить репозиторий сервиса к переиспользуемым workflows
- Как структурировать специфичные для проекта файлы значений Helm
- Как разделить:
    - Логику шаблона
    - Конфигурацию, специфичную для сервиса
- **Как включить PostgreSQL для stateful приложений**

Эти примеры **не являются готовыми к продакшену приложениями**.  
Они служат **референсной реализацией** для интеграции с CI/CD шаблоном.

---

## ⚙️ Как работают примеры

Каждый демонстрационный сервис содержит:

### 1. Код приложения

Минимальная реализация сервиса вместе с Dockerfile.

### 2. Конфигурация развертывания

Директория `deploy/` с многоуровневыми файлами значений Helm:

- `values.yaml` — базовая конфигурация
- `values-dev.yaml` — переопределения для разработки
- `values-prod.yaml` — переопределения для продакшена

### 3. Легковесные workflows

Директория `.github/workflows/`, которая вызывает переиспользуемые workflows из репозитория шаблона.

### 4. Опциональная конфигурация PostgreSQL

Чтобы включить PostgreSQL, обновите файл `values.yaml` следующим образом:

```yaml
postgresql:
  enabled: true
  username: your-username
  password: your-password
  database: your-database
```

## ⚙️ Основные параметры

| Параметр | Описание |
|----------|------------|
| component_name | Название сервиса |
| deploy_mode | ssh / kubernetes |
| image_name | Docker-образ |
| release_name | Релиз Helm |
| namespace_prod | Пространство имен для продакшена |
| namespace_dev | Пространство имен для разработки |
| values_files | Файлы значений Helm |
| build_args | Аргументы сборки Docker |
| security_enabled | Включить этап security-сканирования |
| security_fail_on_findings | Падать на найденных проблемах (блокирует деплой) |
| security_tools | Список инструментов (например, `trivy,osv`) |
| trivy_version | Версия Trivy (зафиксированная) |
| trivy_severity | Уровни критичности |
| trivy_ignore_unfixed | Игнорировать unfixed уязвимости |

## 🔐 Необходимые секреты

### Контейнерный реестр

- GHCR_USERNAME
- GHCR_PAT

### Доступ к Kubernetes

- CLOUDFLEET_ACCESS_TOKEN_ID
- CLOUDFLEET_ACCESS_TOKEN_SECRET
- CLOUDFLEET_ORGANIZATION_ID
- CLOUDFLEET_CLUSTER_ID

### Секреты для сборки frontend (опционально)

- NEXT_PUBLIC_GOOGLE_CLIENT_ID
- NEXT_PUBLIC_GOOGLE_MAPS_API_KEY

Секреты, необходимые для сборки фронтенда, теперь можно передавать напрямую через `BUILD_ARGS`. Это упрощает процесс и обеспечивает безопасную обработку конфиденциальных данных во время сборки.

#### Пример использования

Чтобы передать секреты в качестве аргументов сборки, вы можете определить их в вашем CI/CD пайплайне или напрямую в команде сборки. Пример настройки `BUILD_ARGS` в YAML-файле:

```yaml
build_args:
  NEXT_PUBLIC_GOOGLE_CLIENT_ID: ваш-client-id
  NEXT_PUBLIC_GOOGLE_MAPS_API_KEY: ваш-api-key
```

Эти аргументы будут переданы в процесс сборки Docker и сопоставлены с переменными окружения внутри контейнера. Убедитесь, что в `Dockerfile` добавлены следующие инструкции для обработки этих аргументов:

```dockerfile
ARG NEXT_PUBLIC_GOOGLE_CLIENT_ID="1234567890.apps.googleusercontent.com"
ARG NEXT_PUBLIC_GOOGLE_MAPS_API_KEY="AIzaSyD-EXAMPLEKEY"
ENV NEXT_PUBLIC_GOOGLE_CLIENT_ID=$NEXT_PUBLIC_GOOGLE_CLIENT_ID
ENV NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=$NEXT_PUBLIC_GOOGLE_MAPS_API_KEY
```

Такая настройка гарантирует, что секреты будут доступны как переменные окружения во время выполнения.

## 🔧 Как адаптировать для вашего проекта

1. Замените пример кода приложения на ваш собственный
2. Сохраните обертки для workflows и обновите:
    - Названия образов
    - Названия релизов
    - Пространства имен
    - Пути к файлам значений
3. Настройте файлы значений Helm
4. Настройте секреты репозитория
5. Запустите pipeline на `develop` и `main`
